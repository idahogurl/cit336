Starting Your Work

Introduction

We will be using Git to submit assignments using branches and merge requests. You will not submit anything in iLearn with the exception of the weekly report.

To use Git, we have GitLab hosting our repositories. GitLab, Bitbucket, GitHub are the most well known Git hosts. Git is a version control system. For those unfamiliar with Git, here is a video that gives a high-level explanation.




So, Git provides:

  1. the ability for multiple individuals to work on the same code
  2. a timeline of saves called commits, where each has a description and date

Branching

The way Git implements the first functionality is through the use of branches. When a repository is created, a branch called "master" is automatically created. From "master", individuals create new branches. Sometimes branches are discarded, but usually, the branch gets merged into "master" by the merge request approver once approved.

Sometimes you cannot merge because of competing changes (your changes vs. another person's changes to the same file) that Git cannot resolve without help. This is called a merge conflict. Since you are the only developer working on your code this should rarely happen. If it does happen, please contact your instructor for help. Resolving merge conflicts is beyond the scope of this course.

Before you start an enhancement, you must create a branch as outlined in the video and steps below. As you work, you will commit your changes to that branch.

If you want to learn more about branching and merging, you can read this article.

Committing and Pushing Changes

As you work, you should make periodic commits. A commit is when your file changes are saved to your local repository. Unlike saving to your computer, you do not need to commit for every little change. Commit when you have a portion of work done. "Add stylesheet for homepage" or "Fix hero image for tablet" are good examples of commit messages. Remember a commit is local until you "push" it to your Git host.

Submitting Your Work

Merge Requests

A Merge Request is merely a request to merge your branch of code changes into another branch; usually, this will be the "master" branch. GitHub calls them a "Pull Request."

Merge requests give the ability for another individual to review and discuss the proposed changes. You will have your code changes reviewed by your instructor, who will then merge your branch into the "master" branch. The creation date of the merge request serves as your assignment submission date. If your merge request is created after the due date then it is late.

You must have your commits pushed before you create your merge request. Unless the merge request is locked, you don't have to create a new merge request if you need to add changes prior to the due date. Since commits are dated your instructor will know if you try to sneak in changes before its graded.

Do not wait for your merge request to be merged into "master" for you to start the next assignment. Just create a new branch from the branch of your last enhancement, and you are ready to go. At the end of the semester, you should have 11 branches since we won't be deleting branches after merge.

For every enhancement, you will follow the branch and merge request steps. You will create a branch that you will commit to as you work. When you've finished committing all your changes then create a merge request to submit it.
